# Review: 11. 安全、权限与合规

## Overall Verdict: Pass

该文档达到「强技术型 Agent 产品经理读完后达到 80% 面试可用理解」标准。它覆盖了 Agent 安全中最关键的产品问题：prompt injection、data leakage、越权工具调用、最小权限、human approval、审计日志、回滚、安全评测、GTM/Sales Agent 场景和合规参考。

整体判断：可以作为面试准备材料使用。文档的强项是把安全从“提示词防御”提升到“数据、权限、工具、行动、审计、评测”的系统设计层面，符合 Agent PM 面试中对产品判断和工程沟通能力的要求。主要提升空间在于：合规要求可以更明确映射到产品能力，MCP/第三方工具供应链风险可以更具体，企业管理员控制台和 incident response 流程可以再产品化一些。

## Top Issues Ranked By Severity

1. Medium: 合规内容覆盖面好，但产品映射还可以更明确。
   文档提到 GDPR、EU AI Act、SOC 2、ISO 27001、DPA、数据驻留、零数据保留，也引用了可靠来源。但正文主要讲“要知道这些概念”，没有把它们逐项映射到 Agent 产品能力和销售/采购证据。例如：GDPR 对 data minimization、retention、DPA、data subject rights 的产品影响；EU AI Act 对 high-risk 场景的日志、人类监督、文档和鲁棒性要求；SOC 2/ISO 27001 更偏组织控制和证据，不等同于单个 Agent 功能。

2. Medium: MCP/connector/第三方工具供应链风险还可以更尖锐。
   文档引用了 OWASP MCP Top 10，也在 PM 可交给工程深挖处提到 MCP server 安全审查和 tool schema fuzzing，但正文没有系统解释 connector 场景的产品风险：tool description poisoning、工具能力变更、过宽 OAuth scope、第三方 connector trust、token/secret 暴露、cross-tool data exfiltration、用户批准后工具行为变化等。2026 Agent PM 面试里，这类问题很容易被追问。

3. Medium: 缺少企业管理员控制台和策略配置的具体产品形态。
   文档多次说 PM 应产出管理员控制台需求，但没有展开控制台应包含哪些功能。对于 B2B Agent，安全权限常常要通过 admin UX 落地，例如工具启停、角色策略、审批规则、日志检索、数据保留、模型/供应商开关、区域策略、connector scopes、风险告警和导出报告。补上这部分会更贴近企业采购和落地。

4. Low: Incident response 和持续治理略偏概念。
   文档提到事故复盘、告警、回滚和 incident_id，但没有给出安全事件发生后的 PM 版流程：检测、冻结能力、撤销 token、通知管理员、定位 trace、数据影响评估、客户沟通、策略热修复、eval 回归、postmortem。这不是通过门槛问题，但会提升高级面试表现。

5. Low: 示例很强，但缺少一条完整攻击链 trace 与审计日志样例。
   现有例子覆盖风险和控制，但如果加入一条简短的 end-to-end trace，如“外部网页注入 -> extraction node -> policy block -> approval denied -> audit log fields”，读者会更容易回答“系统里怎么证明你拦住了”的追问。

## Missing Topics

- 合规到产品能力的映射表：GDPR、EU AI Act、SOC 2、ISO 27001、DPA、数据驻留、zero data retention 分别要求 PM 设计或提供什么。
- MCP/connector 风险：tool poisoning、tool description poisoning、OAuth scope 过宽、第三方工具变更、connector 供应链、token/secret 保护、cross-tool exfiltration。
- 企业管理员控制台：工具 allowlist、角色/策略配置、审批规则、日志与导出、数据保留、模型供应商策略、告警、connector 权限审查。
- Incident response playbook：发现异常后如何停用能力、撤销授权、评估影响、通知客户、修复策略和回归评测。
- 更具体的安全 launch checklist：beta、pilot、GA 各阶段哪些能力允许自动化，哪些指标必须达标。
- 合规敏感行业差异：金融、医疗、教育、公共部门、跨境销售团队在 Agent 权限和数据处理上的不同要求。
- 完整审计日志样例或 trace 示例，帮助读者把表格字段转化为可讲述的系统证据。

## Suggested Edits

- Add a subsection: “合规要求如何变成 Agent 产品能力.”
  建议做表格：法规/认证/合同要求、PM 应设计的能力、销售或安全审查时可展示的证据。示例：GDPR -> 数据最小化、保留期、删除/导出请求、DPA；EU AI Act -> 日志、人类监督、风险管理、鲁棒性；SOC 2 -> 访问控制、审计证据、变更管理、incident process。

- Add a subsection: “MCP / connector security risks.”
  用 GTM Agent 的 CRM、邮箱、营销平台 connector 举例，说明 connector scope、tool description、token storage、tool update、第三方信任边界和跨工具数据外泄。保持 PM 深度，不需要工程实现细节。

- Expand the enterprise admin console requirement.
  建议列出控制台功能：启用/禁用工具、按角色配置 read/write/send/export 权限、设置审批阈值、查看审计日志、配置数据保留和脱敏、管理模型供应商、查看安全评测结果、导出合规报告。

- Add an incident response mini-playbook.
  建议用 8 步：detect -> contain -> revoke -> investigate -> assess impact -> notify -> patch/eval -> postmortem。说明 PM 在每一步要定义的用户体验、客户沟通和产品开关。

- Add one concrete trace example.
  例如：外部网页含 prompt injection，Agent 抽取结构化字段时标记 `injection_flags=true`，policy engine 阻止 `crm.export_contacts` 和 `email.send`，审计日志记录 blocked action、source URL、risk label、policy version。

- Add sharper interview follow-up answers.
  可补 3 个追问：客户要求全自动发送邮件怎么办；安全团队要求保存完整 prompt 怎么平衡；如何向采购证明 Agent 不会越权访问 CRM。

## 80% Interview-Ready Judgment

Yes. The document reaches approximately 85-90% interview-ready understanding for a strong technical Agent PM.

After reading it, a PM should be able to:

- Explain why Agent security is harder than chatbot safety.
- Distinguish direct and indirect prompt injection and explain why external content is untrusted.
- Describe data leakage paths across prompt, RAG, memory, tool args, tool results, logs and final output.
- Design least-privilege permissions using user identity, tenant, resource, field, action, scenario, time, quantity and network boundaries.
- Explain when human approval is required and what a good approval UX must show.
- Define audit log fields, rollback strategy and safety eval metrics.
- Tell a credible GTM/Sales Agent safety story in interviews.

The remaining gap is mostly senior-level productization detail: compliance evidence, connector governance, admin controls and incident response. These are important polish areas, but they do not prevent the document from passing the stated 80% interview-readiness threshold.
