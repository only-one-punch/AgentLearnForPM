04 / Tool Calling 工具调用

Tool Calling 如何让 Agent 从“会聊天”变成“能执行任务”

面向 Agent PM 的理解重点，不是记住某个 SDK 的写法，而是判断什么时候需要工具、工具边界在哪里、 参数和权限如何控制，以及失败时产品应该如何兜底。

核心概念

Tool Calling 是让模型输出结构化调用意图，并由系统执行外部工具的机制。工具可以是查询数据库、 调用搜索、创建工单、发送消息，也可以是内部业务系统中的任意动作。

对产品经理来说，Tool Calling 的价值是把模型能力接到真实业务动作上；风险是模型判断、参数生成和工具执行都可能出错。

典型流程

一个最小流程通常包括：用户提出目标，模型判断是否需要工具，模型生成工具名和参数，系统执行工具， 工具返回结果，模型基于结果继续推理或输出最终答案。 

用户目标
  -> 模型判断需要哪个工具
  -> 生成结构化参数
  -> 工具执行
  -> 返回结果
  -> 模型继续推理或输出

PM 需要判断什么

PM 需要判断这个动作是否适合自动化，失败成本是否可接受，是否需要人工确认，工具权限是否过大， 以及结果是否能被用户理解和纠正。

  • 低风险查询类工具可以更自动化。
  • 高风险写入类工具需要确认、撤销或审批。
  • 不稳定工具需要清楚的失败提示和替代路径。

产品边界

如果 Agent 要给销售生成线索，它可以自动查公司、找关键人、总结痛点和生成触达理由； 但是否直接发送消息，取决于场景风险、品牌风险和销售团队的控制需求。
